RF1
Requerimiento Funcional 1 · 20%

Los formularios del sistema 20%

Antes de guardar cualquier dato, el sistema necesita saber exactamente qué información espera recibir y en qué formato. Para eso creamos los modelos Pydantic: formularios prediseñados que el sistema conoce de memoria.

Analogía del día a día

Imagina que vas a hacerte un examen médico. La enfermera te entrega un formulario con campos específicos: nombre (solo letras), edad (solo números), tipo de sangre (solo A, B, AB u O). No puedes escribir "mañana" en el campo de edad ni dejar el nombre en blanco. El formulario sabe qué esperar y rechaza lo que no tiene sentido. Eso es exactamente lo que hacen nuestros modelos.

Creamos 3 formularios (modelos) anidados, es decir, uno dentro del otro, como una carpeta que contiene varias hojas:

👤
Formulario del Encuestado
Datos de la persona: nombre, edad, estrato (1–6), departamento de Colombia, género, ingresos. Cada campo tiene su tipo de dato permitido.
Encuestado
❓
Formulario de Respuesta
Cada pregunta contestada: el texto de la pregunta, su tipo (escala, porcentaje, texto libre) y la respuesta. La respuesta puede ser número o texto según el tipo.
RespuestaEncuesta
📋
Formulario Completo
El paquete completo: contiene el formulario del Encuestado más una lista de todas sus respuestas. Es lo que se envía al sistema de una sola vez.
EncuestaCompleta
pythonmodels.py — así luce el formulario en código
# Un "formulario" en Python: dice qué campos espera y de qué tipo
class Encuestado:
    nombre:   str            # texto obligatorio
    edad:     int            # número entero obligatorio
    genero:   Optional[str]  # texto, pero OPCIONAL (puede ir vacío)
    estrato:  int            # número del 1 al 6

# La respuesta puede ser número entero, decimal, texto o vacío
# → Union = "acepto cualquiera de estos tipos"
    respuesta: Union[int, float, str, None]

# El formulario completo: un Encuestado + LISTA de respuestas
class EncuestaCompleta:
    encuestado: Encuestado             # formulario anidado dentro de otro
    respuestas: List[RespuestaEncuesta] # lista de formularios de respuestas
✅
¿Por qué es esto importante?

Si alguien intenta enviar una encuesta con edad = "mucho" o estrato = 99, el sistema lo rechaza automáticamente antes de guardar nada. Los datos que entran al repositorio siempre son limpios y consistentes.

RF2
Requerimiento Funcional 2 · 20%

La aduana de datos 20%

Los formularios saben qué tipo de dato esperar, pero los validadores van más allá: revisan que el dato tenga sentido en el contexto real colombiano.

Analogía del día a día

En el aeropuerto, la aduana no solo revisa que tengas pasaporte (que el documento exista), sino que no esté vencido, que seas tú la persona de la foto, que no traigas productos prohibidos. Esas son reglas adicionales sobre datos que ya existen. Nuestros validadores hacen lo mismo con los datos de las encuestas.

🗺️
Validador de Departamento
Verifica que el departamento sea uno de los 33 oficiales de Colombia. Si alguien escribe "Narnia" o "California", el sistema devuelve un error claro. Además, acepta el nombre en minúsculas ("antioquia") y lo convierte a mayúsculas ("ANTIOQUIA") antes de guardarlo. Este es un validador mode='before': actúa antes de que el sistema procese el dato — como un traductor que normaliza el idioma antes de pasarlo al inspector.
🏢
Validador de Estrato
Colombia tiene estratos del 1 al 6. Si alguien envía estrato 7, 0 o -1, el sistema lo rechaza con un mensaje claro. Este es un validador mode='after': actúa después de confirmar que el dato ya es un número entero, solo revisando la regla de negocio colombiana — como el inspector que ya sabe que es un pasaporte válido y ahora solo revisa si no está vencido.
📊
Validador de Respuesta según Tipo de Pregunta
Si una pregunta es de escala Likert, la respuesta debe ser un número del 1 al 5. Si es de porcentaje, debe estar entre 0 y 100. Este validador cruza dos campos a la vez. Es como si un profesor revisara que las respuestas de opción múltiple sean A, B, C o D, y no "sí" o "no".
📅
Validador de Fecha
La fecha debe tener el formato YYYY-MM-DD. Si alguien escribe "20 de marzo" o "20/03/26", el sistema rechaza y explica el formato correcto. Como cuando un formulario del banco dice: "Ingrese la fecha como DD/MM/AAAA".
🚫
Validador de Preguntas Duplicadas
Una encuesta no puede tener dos respuestas para la misma pregunta. Si alguien envía la pregunta P001 dos veces, el sistema detecta la duplicación y la rechaza. Como si en un formulario físico no se pudiera responder dos veces la misma pregunta.
💡
La diferencia entre before y after con un ejemplo concreto

mode='before' es como revisar y organizar tu maleta antes de que entre a la radiografía del aeropuerto: puedes acomodar las cosas, sacar lo que no debe ir, cambiar la presentación. mode='after' es como revisar lo que ya salió de la radiografía: el contenido ya está confirmado, solo verificas si trae algo que no debería.

Requerimiento Funcional 3 · 20%

Las ventanillas del banco 20%

La API es como una oficina con diferentes ventanillas: cada una tiene un propósito distinto y atiende un tipo de solicitud diferente.

Analogía del día a día

En un banco hay ventanillas para depositar dinero, consultar saldo, actualizar datos personales, o cancelar una cuenta. No puedes depositar en la ventanilla de consultas ni al revés. Nuestra API funciona igual: cada endpoint es una ventanilla con una función específica, y el tipo de solicitud HTTP indica qué quieres hacer.

AcciónVentanilla (ruta)¿Qué hace?Respuesta
POST/encuestas/📥 Registrar encuesta nueva201 Creado
GET/encuestas/📋 Ver todas las encuestas200 OK
GET/encuestas/estadisticas/📊 Ver resumen estadístico200 OK
GET/encuestas/{id}🔍 Ver una encuesta por código200 / 404
PUT/encuestas/{id}✏️ Corregir encuesta existente200 / 404
DELETE/encuestas/{id}🗑️ Eliminar una encuesta204 / 404
⚠️
Un detalle importante: el orden de las ventanillas

La ventanilla de /estadisticas/ se registró antes que la ventanilla /{id} en el código. ¿Por qué importa? Porque el sistema lee las rutas en orden. Si /{id} fuera primero, cuando alguien pida /estadisticas/, el sistema pensaría que "estadisticas" es un código de encuesta y respondería con error. Es como poner un letrero específico antes del letrero genérico en los pasillos del hospital.

Además del CRUD básico, el sistema tiene 3 ventanillas de carga masiva:

📁
Desde un archivo
Acepta CSV, Excel, JSON, Parquet, TXT y más. Como enviar un formulario físico escaneado, pero digital.
🌐
Desde una URL de nube
Se conecta a Google Drive, Dropbox u OneDrive y descarga el archivo. Como pedir que te envíen el documento por correo.
🔌
Desde otra API
Se conecta a otro sistema externo y trae los datos. Como llamar a otra oficina para que le pasen la información.

Las cartas de rechazo explicadas 10%

Cuando alguien envía datos incorrectos, el sistema no puede simplemente decir "no". Necesita explicar exactamente qué falló y por qué.

Analogía del día a día

Imagina que presentas una solicitud de crédito y te la rechazan. Una respuesta pésima: "Solicitud rechazada". Una respuesta útil: "Su solicitud fue rechazada por tres razones: 1) El número de cédula tiene 9 dígitos pero debe tener 10. 2) El salario mensual no puede ser negativo. 3) La ciudad de residencia no es válida." Nuestro sistema devuelve exactamente ese tipo de respuesta detallada.

jsonRespuesta cuando los datos son incorrectos (HTTP 422)
{
  "estado": "error_validacion",
  "mensaje": "Los datos tienen errores. Revise cada campo.",
  "errores": [
    {
      "campo":          "encuestado → estrato",
      "mensaje":        "Estrato '9' inválido. En Colombia los estratos son del 1 al 6.",
      "valor_recibido": 9
    },
    {
      "campo":          "encuestado → departamento",
      "mensaje":        "'Narnia' no es un departamento válido de Colombia.",
      "valor_recibido": "Narnia"
    }
  ]
}
📝
Auditoría automática: el libro de registro

Cada vez que alguien envía datos incorrectos, el sistema lo registra automáticamente en la consola del servidor con fecha, hora y detalle del error. Es como un libro de registro en la recepción: queda constancia de cada intento fallido, lo que sirve para detectar errores frecuentes o intentos malintencionados.

Requerimiento Funcional 5

El mesero eficiente RF3

Los endpoints del sistema usan programación asíncrona: una forma de atender muchas solicitudes al mismo tiempo sin hacer esperar a nadie innecesariamente.

Analogía del día a día

Imagina dos tipos de meseros. El síncrono toma el pedido de la mesa 1, va a la cocina, espera parado a que el chef prepare la comida (10 minutos), regresa a entregar el plato, y solo entonces atiende a la mesa 2. Si hay 10 mesas, la última espera 100 minutos. El asíncrono toma el pedido de la mesa 1, lo entrega en cocina, y mientras la comida se prepara atiende a las mesas 2, 3 y 4. Cuando la cocina avisa que el plato está listo, regresa a entregarlo. Puede atender 50 mesas donde el otro solo podría atender 5.

🐢 Síncrono (def)
  • Atiende una solicitud a la vez
  • Mientras espera respuesta de internet, bloquea todo lo demás
  • 10 usuarios simultáneos = 10 solicitudes en cola
  • Útil solo cuando el trabajo es cálculo puro sin esperas de red
⚡ Asíncrono (async def)
  • Maneja miles de solicitudes al mismo tiempo
  • Mientras espera de internet, atiende otras solicitudes
  • Indispensable al descargar archivos de Google Drive o consultar otra API
  • El modelo que usa FastAPI con Uvicorn (ASGI)
🏗️
¿Qué es ASGI y por qué importa?

ASGI es el estándar sobre el que corre FastAPI, implementado por Uvicorn (el servidor). La alternativa más antigua, WSGI, funciona como el mesero síncrono: un proceso por solicitud. ASGI funciona como el mesero asíncrono: un solo proceso maneja miles de solicitudes simultáneas delegando las esperas al sistema operativo. Para una API que descarga archivos o consulta otras APIs, esto marca una diferencia enorme.

La lonchera del colegio 15%

El proyecto corre dentro de un entorno virtual: una copia aislada de Python con solo las herramientas que este proyecto necesita, sin mezclar con otros proyectos del computador.

Analogía del día a día

Imagina que tienes una mochila diferente para cada materia: una para matemáticas con calculadora y compás, otra para arte con pinceles y acuarelas. Si mezclas todo en una sola mochila, los pinceles manchan los cuadernos y la calculadora termina llena de pintura. El entorno virtual es la mochila separada para cada proyecto de programación: cada uno tiene exactamente lo que necesita, sin interferir con los demás.

bashCrear y activar la mochila del proyecto
# 1. Crear una "mochila" exclusiva para el proyecto
python -m venv .venv

# 2. "Abrir" la mochila (activar el entorno)
source .venv/bin/activate    # Mac / Linux
.venv\Scripts\activate       # Windows

# 3. Instalar todo lo que el proyecto necesita
pip install -r requirements.txt
🤔
¿Por qué venv y no conda?

Usamos venv porque viene incluido en Python sin instalar nada extra, y es suficiente para este proyecto que es 100% Python. conda sería útil si necesitáramos manejar diferentes versiones de Python o instalar librerías con partes en otros lenguajes como C o R — como tener una mochila con compartimentos especiales para materiales frágiles. Aquí no es necesario.

RT5
Requerimiento Técnico 5 · 10%

Los ayudantes silenciosos 10%

Los decoradores son ayudantes que se añaden a una función para que hagan algo extra de forma automática, sin modificar el código original de la función.

Analogía del día a día

Imagina que un periodista graba una entrevista. El camarógrafo no interfiere con la entrevista, no hace preguntas ni responde — pero registra todo lo que pasa. El cronómetro tampoco interfiere, solo mide el tiempo. Ambos son "ayudantes silenciosos" que trabajan alrededor de la entrevista sin cambiar su contenido. Así funcionan nuestros decoradores: @log_request es el camarógrafo y @timer es el cronómetro.

📹
@log_request — El camarógrafo
Cada vez que se llama un endpoint, registra automáticamente en la consola: la hora exacta, qué función se ejecutó y cuánto tardó. Si algo falla, registra el error. Todo sin que el endpoint tenga que hacer nada especial.
⏱️
@timer — El cronómetro
Mide el tiempo de ejecución con precisión de milisegundos. Se usa en estadísticas para saber cuánto tarda el cálculo. Útil para detectar operaciones lentas antes de que sean un problema.
pythonmain.py — así se usan los decoradores apilados
# El endpoint de estadísticas usa DOS ayudantes a la vez:
@app.get("/encuestas/estadisticas/")  # FastAPI registra la ruta
@log_request   # el camarógrafo entra primero
@timer         # el cronómetro también
async def estadisticas():
    # Esta función no sabe que tiene ayudantes.
    # Solo hace su trabajo: calcular estadísticas.
    return services.calcular_estadisticas()
🔗
¿Cómo se relacionan con @app.get y @app.post?

Los decoradores de FastAPI (@app.get, @app.post) son decoradores de registro: le dicen al servidor qué función atiende qué ruta. Nuestros decoradores (@log_request, @timer) son de comportamiento: agregan funcionalidad extra sin modificar la lógica. Todos usan el mismo mecanismo de Python — como diferentes tipos de etiquetas que se pueden poner sobre la misma función, cada una con un propósito distinto.

Guía de ejecución

Cómo poner el proyecto en marcha

Descomprimir la carpeta
Extraer el archivo encuesta-api.zip y abrir la carpeta. Todos los archivos del proyecto deben estar juntos en un mismo lugar.
Crear y activar el entorno virtual
Ejecutar python -m venv .venv y luego source .venv/bin/activate (Mac/Linux) o .venv\Scripts\activate (Windows). El terminal mostrará (.venv) al inicio cuando esté activo.
Instalar las herramientas
Ejecutar pip install -r requirements.txt. Descarga e instala todo lo necesario automáticamente. Como ir al supermercado con una lista de compras exacta.
Iniciar el servidor
Ejecutar python main.py. El sistema buscará automáticamente un puerto libre y mostrará la dirección. Abrir esa dirección en el navegador y el sistema estará listo.
🎲
¿Por qué el número de puerto cambia cada vez?

El sistema no usa un número fijo como 8000. En cambio, busca un puerto que esté realmente libre en el computador, verificándolo con una prueba de red. Es como buscar un estacionamiento disponible en lugar de ir siempre al mismo que puede estar ocupado. Así el servidor siempre arranca sin conflictos.
